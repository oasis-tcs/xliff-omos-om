Abstract:

This document defines version 1.0 of the Localization Interchange Object Model (LIOM) based on the XLIFF 2 Family of Standards. The purpose of this Object Model is to provide a serialization independent definition of objects, their relationships and data models, so that Localization tools can exchange translation and localization task data in an interoperable way without mandating a specific serialization of the Object Model. Specific serialization and binding provisions are out of scope of this specification, albeit XLIFF (the original XML serialization), JLIFF (JSON Localization Interchange Fragment Format), and other hypothetical serializations examples are used to demonstrate the semantic and behavioral match between and among different serialization formats as facilitated by this Object Model. The ultimate goal of this specification is to enable localization tools to pass localizable content including software strings from one step of the localization process to any other, while enabling interoperability between and among tools without enforcing a specific serialization.

1 Introduction

Localization Interchange Object Model (LIOM) is the XLIFF 2 Object Model designed by a group of multilingual content publishers, software providers, localization service providers, localization tools providers and researchers. It is intended to give any multilingual content owner a single object and data model that can be subject to any sort of serialization common now (such as XML, JSON, or protobuf) or a in the future. XLIFF 2 has been providing a reliable XML based interchange file format understood by any localization provider, using any conformant localization tool. The primary focus of this Object Model is to ensure lossless Interchange between and among pipelines based on different serializations and technologies. While the primary focus is on lossless Interchange, toolmakers, service providers, and content owners are encouraged to stick to this Object Model also while designing their internal processing and messaging formats, as this will make the mapping to the standardized Interchange format serializations of this Object Model, such as JLIFF or XLIFF, so much easier.

1.1 Terminology

1.1.1 Key words

1.1.2 Definitions

Agent

any application or tool that generates (creates), reads, edits, writes, processes, stores, renders or otherwise handles documents, messages, or any other instantiations of the LIOM.

Agent is the most general application conformance target that subsumes all other specialized user agents disregarding whether they are defined in this specification or not.

Enrich, Enriching

the process of associating metadata and resources with the Extracted LIOM based payload

Processing Requirements

  • Enriching MAY happen at the time of Extraction.

Note

Extractor knowledge of the native format is not assumed while Enriching.

Enricher, Enricher Agent

any Agent that performs the Enriching process

Extract, Extraction

the process of encoding localizable content from a native content or User Interface format as LIOM based payload, so that localizable parts of the content in the source language are available for Translation into the target language along with the necessary context information

Extractor, Extractor Agent

any Agent that performs the Extraction process

Merge, Merging

the process of importing LIOM based payload back to the originating native format, based on the full knowledge of the Extraction mechanism, so that the localized content or User Interface strings replace the source language in the native format

Merger, Merger Agent

an Agent that performs the Merge process

Warning

Unless specified otherwise, any Merger is deemed to have the same knowledge of the native format as the Extractor throughout the specification.

Mergers independent of Extractors can succeed, but it is out of scope of this specification to specify interoperability for Merging back without the full Extractor knowledge of the native format.

Modify, Modification

the process of changing core and module LIOM based structural and inline elements that were previously created by other Writers

Processing Requirements

  • LIOM objects MAY be Modified and Enriched at the same time.

Note

Extractor or Enricher knowledge of the native format is not assumed while Modifying.

Modifier, Modifier Agent

an Agent that performs the Modification process

Translation, Translate

a rendering of the meaning of the source text, expressed in the target language

Writer, Writer Agent

an Agent that creates, generates, or otherwise writes a document or meXLIFF Document for whatever purpose, including but not limited to Extractor, Modifier, and Enricher Agents.

Note

Since XLIFF is intended as an exchange format rather than a processing format, many applications will need to generate XLIFF Documents from their internal processing formats, even in cases when they are processing XLIFF Documents created by another Extractor.

1.1.3 Key concepts

LIOM Core

The core of LIOM 1.0 consists of the minimum set of data objects required to (a) prepare a LIOM Instance that contains text Extracted from one or more files for localization, (b) allow it to be completed with the translation of the Extracted text, and (c) allow the generation of Translated versions of the original document.

The XML namespace that corresponds to the core subset of both [XLIFF 2.1] and [XLIFF 2.0] is "urn:oasis:names:tc:xliff:document:2.0".

XLIFF-defined (elements and attributes)

The following is the list of allowed schema URI prefixes for XLIFF-defined elements and attributes:

urn:oasis:names:tc:xliff:
https://www.w3.org/2005/11/its/

However, the following namespaces are NOT considered XLIFF-defined for the purposes of the XLIFF 2.1 specification:

urn:oasis:names:tc:xliff:document:1.0
urn:oasis:names:tc:xliff:document:1.1
urn:oasis:names:tc:xliff:document:1.2
urn:oasis:names:tc:xliff:changetracking:2.0

Elements and attributes from other namespaces are not XLIFF-defined.

Source: [XLIFF 2.1].

Because the definition of XLIFF-defined has been updated in [XLIFF 2.1], see [XLIFF 2.0] for an XLIFF 2.0 conformant definition.

Note

Implementers of the LIOM need to parametrize XLIFF-defined to achieve full backwards and forwards compatibility with any XLIFF 2.x specifications.

LIOM-defined namespaces

All XLIFF-defined namespaces are LIOM-defined namespaces. This specification doesn't define any LIOM-defined namespaces that are not XLIFF-defined, however the notions are not considered strictly equivalent and future LIOM versions MAY define LIOM-defined namespaces that are not XLIFF-defined. Albeit it would be expected that future XLIFF Versions SHOULD adopt those or at least should not adopt other namespaces with incompatible semantics.

LIOM Conformant Specification

A publicly available specification that normatively defines a serialization (document or message payload format) that conforms to the LIOM defined in this specification.

Currently known conformant specifications include XLIFF Version 2.0 [XLIFF 2.0], XLIFF Version 2.1 [XLIFF 21], and JLIFF Version 1.0.

LIOM Instance

A document or message payload that is a valid instance of any LIOM Conformant Specification.

Note

Documents or messages cannot conform directly to LIOM, as LIOM doesn't define a serialization and an ad hoc format cannot be reliably checked against the complex requirements of the LIOM. Currently known LIOM instances include XLIFF 2 files or JLIFF Instances. XLIFF 1.2 files are NOT LIOM Instances.

LIOM Module

A module is an OPTIONAL set of data and metadata categories that were designed to store information about a process applied to a LIOM Instance and the data incorporated into a LIOM Instance as result of that process.

Each official module defined for LIOM Version 1.0 has its data categories and structure defined within this specification and corresponds to at least one dedicated namespace, see Normatively Used Namespaces.

2 The Core Specification

LIOM is a bilingual object model designed for containing text that needs Translation, its corresponding translations and auxiliary data that makes the Translation process possible.

At creation time, a LIFF Object MAY contain only text in the source language. Translations expressed in the target language MAY be added at a later time.

The LIOM postulates several possible root objects. The highest possible level root object is LIF.

The LIF object corresponds to the root element of an XLIFF Document, which is <xliff>.

WORK IN PROGRESS!!! THE FOLLOEING PARTS OF THE SPECIFICATION NEED RESTATED IN AN XML INDEPENDENT WAY It contains a collection of <file> elements. Typically, each <file> element contains a set of <unit> elements that contain the text to be translated in the <source> child of one or more <segment> elements. Translations are stored in the <target> child of each <segment> element.

2.1 General Processing Requirements

  • An Agent processing a valid LIOM Instance that contains only objects from namespaces that correspond to XLIFF-defined elements and attributes that it cannot handle MUST preserve those elements and attributes.

  • An Agent processing a valid LIOM Instance that contains custom (extension objects from 3rd party namespaces) objects that it cannot handle SHOULD preserve those objects and values.
